# 用户体验优化与功能缺失分析报告

**日期**: 2026-01-20
**状态**: 待执行

本文档基于真实用户视角对 Gepei App V1.2 版本进行了深度体验审查，梳理出关键的 UX 断层和业务流程缺失点。

## 1. 核心体验问题 (UX Issues)

### 1.1 地陪资料管理流程断层
**严重程度**: 高
**问题描述**:
1.  **无反馈**: 用户在 `GuideEdit` 页面提交修改后，虽然有 Toast 提示，但页面停留在编辑态，用户不知道操作是否生效，甚至可能重复点击保存。
2.  **入口混淆**: 在 `Profile` 页面，无论用户是否已经是地陪，按钮文案始终为“认证为地陪”。对于已认证的地陪，缺乏明确的“修改资料”入口。
3.  **回显体验**: 首次进入编辑页时，如果网络稍慢，表单可能显示空白，缺乏加载骨架屏。

**优化建议**:
- **提交后跳转**: 保存成功后，自动跳转回 `/profile` 页面。
- **动态入口**: `Profile` 页面根据 `user.is_guide` 状态，显示“认证为地陪”或“管理地陪资料”。
- **完善回显**: 确保编辑页在数据加载完成前显示 Loading 状态，避免用户看到空白表单。

### 1.2 导航与空状态
**严重程度**: 中
**问题描述**:
1.  **空列表**: 当“我的订单”或“地陪列表”为空时，只显示空白或简单的文字，缺乏引导性。
2.  **返回逻辑**: 部分二级页面（如订单详情）的返回按钮逻辑可能不统一（有的用 `history.back()`，有的固定跳首页）。

**优化建议**:
- 增加 Empty State 插画组件。
- 统一使用路由管理的返回逻辑。

## 2. 业务功能缺失 (Missing Features)

### 2.1 定制订单撮合闭环 (Custom Order Matching)
**严重程度**: 极高 (业务无法闭环)
**问题描述**:
目前 FP-013 仅实现了“用户发布需求”和“管理员查看需求”。**缺失了核心的撮合环节**：
1.  管理员无法将定制需求指派给合适的地陪。
2.  地陪无法看到被指派的单子。
3.  用户无法看到管理员推荐的地陪并进行确认/支付。

**导致后果**: 定制单发布后即进入死胡同，无法转化为实际交易。

**解决方案 (FP-019)**:
1.  **管理员端**: 在后台订单详情页，增加“指派地陪”功能（从地陪库中搜索并关联）。
2.  **用户端**: 定制单详情页增加“候选地陪列表”区域，用户可查看地陪资料并点击“确认选择”。
3.  **状态流转**: 确认后，订单状态流转为 `pending` (待支付)，随后进入标准支付流程。

## 3. 建议执行路线图

建议在 V1.3 版本中优先解决上述问题：

1.  **Week 1**: 集中修复 UX 问题 (1.1 & 1.2)，确保现有功能用得顺手。
2.  **Week 2**: 开发 FP-019 定制撮合功能，打通定制业务闭环。
